# Generator / Reviewer 迴圈

把 MOC v0 推到定稿的 subagent pattern。核心邏輯：**reviewer 找問題 → fixer 全改 → 再 review**，直到 reviewer 回「無問題」。

## Reviewer Subagent Prompt

```
## 契約

**輸入**：
- `<MOC 絕對路徑>`：要 review 的 MOC 檔案路徑
- `<第 N 輪>`：本輪 review 編號（用於「第 2 輪以後避免重複報告同一句」）
- 官方 docs URL（可選）：步驟 D 事實校正產出的來源 URL
- `<前幾輪摘要>`（可選）：格式「第 K 輪：N 項（必改 X / 應改 Y / 可選 Z）」，用於終止條件判斷

**輸出**：符合下方「回報格式」的 markdown
**禁止行為**：不寫入或修改任何檔案；不提供修改後的完整段落（只給改法建議）

任務：對 <MOC 絕對路徑> 做完整 review，找出所有剩餘問題。這是第 N 輪 review。

前幾輪摘要（若有）：
<前幾輪摘要，格式：第 K 輪：N 項（必改 X / 應改 Y / 可選 Z）；無則省略>

## 前置：讀取 MOC 結構標準

先從 MOC 絕對路徑推算 repo root（取 `/content/` 之前的部分），再 Read `<repo_root>/.agents/skills/vault-distill/references/moc-structure.md`，作為 review 維度 5 排版與結構的評估基準。

## Review 維度

1. **內部一致性**：前言對應章節、章節互不矛盾、術語/專名一致
2. **事實準確性**（對照官方來源，若有）：
   （每行一個，無則整段省略或填 N/A）
   <官方 docs URL 1>
   <官方 docs URL 2>
   <官方 docs URL 3>
   ...（視步驟 D 校正結果，可有任意多個）
   檢查：數字、規則、語法、API 介面是否正確
   **若呼叫者填 N/A 或整段省略**：跳過此維度，不要硬找來源比對（事實校正是 SKILL.md 步驟 D 的選擇性步驟，沒做就沒有來源）
3. **wikilink 有效性**：用 Glob 或 Read 確認目標檔案存在
4. **Obsidian vault 規則**（真實來源 content/CLAUDE.md 的「寫入前 Checklist」＋ `scripts/vault-schema.mjs`）：
   - frontmatter 欄位、順序、白名單符合 schema；`updated` 盡量同步今日（不強制）
   - 檔名無空格、用連字號
   - `#` 開頭色碼需反引號包住
   - tags 用 YAML 清單
5. **排版與可讀性**（對照 moc-structure.md 的骨架與「常見錯誤」checklist）：孤兒句、多餘空行、表格欄位對齊、章節順序合理、長度是否需拆子主題
6. **外部連結**：格式正確，URL 實際存在（抽樣最多 2-3 個 WebFetch 檢查，不需全檢）
   WebFetch 失敗處理：
   - 收到 403 / 404 → 標記「連結失效」（必改）
   - DNS / timeout（等待超過約 3 秒無回應）→ 標記「無法驗證」（應改）
   - 不要因為無法驗證就跳過報告

## 重要規則

- 專注實質錯誤、語意歧義、斷連結、事實不符
- 不報告純風格偏好（「某字改某字更順」類的主觀）
- 第 2 輪以後避免重複報告已多次微調的同一句
- 優雅定稿優先：文件整體準確、結構清楚、無斷連結、無事實錯誤即可定稿
- 若前幾輪摘要中連續 2 輪只剩「可選」，且本輪也僅發現可選問題 → 在回報結尾加一行：「建議停止：已連續 N 輪只剩可選項目，ROI 遞減，可詢問用戶是否定稿」

## 回報格式

若完全沒問題：明確單句回應「無問題，MOC 可以定稿」

若有問題：分類列出，每條含
- line 編號
- 目前內容
- 建議改法
- 理由（實質錯誤而非風格）

分類：**必改**（影響正確性）/ **應改**（影響體驗）/ **可選**（小修飾）

不要動手改檔案，只做 review 回報。
```

## Fixer Subagent Prompt

```
## 契約

**輸入**：
- `<MOC 絕對路徑>`：要修改的檔案路徑
- `N 項`：要處理的問題數量
- reviewer 清單：必改 / 應改 / 可選 分組
- `<今日日期 YYYY-MM-DD>`：用於 frontmatter updated 欄位

**輸出**：每項「修改前 → 修改後」的 line 對照
**禁止行為**：
- 不改清單以外的文字（包括不修飾周邊句子、不調整措辭風格、不拆分/合併非問題項目的段落）
- 不新增章節、不改章節順序
- 不改 frontmatter 的 title/created（除非 reviewer 明確指出）

任務：修改 <MOC 絕對路徑>，處理以下 N 項 review 問題。

## Obsidian vault 規則
- 修改 .md 時，frontmatter `updated` 欄位盡量同步為今日日期（不強制）；日期由呼叫者填入 `<今日日期 YYYY-MM-DD>`
- 詳見 content/CLAUDE.md

## 要處理的問題

<貼上 reviewer 回報的完整清單，按必改/應改/可選分組>

## 執行要求

- 必改與應改：全部處理，不跳過
- 可選：跳過（不需特別說明）
- 不新增章節、不改章節順序
- 不改 frontmatter 的 title/created（除非 reviewer 指出）；tags 由 reviewer 明確標記後可改
- 完成後回報每項修改前後的 line 內容

**「無法執行」的定義**（遇到以下情況明確說明，不硬改）：
- 連結其實有效（reviewer 報失效，但 fixer 讀到時連結正常）
- 段落已不存在（被其他操作刪除或移位，與 reviewer 描述的 line 不符）
- reviewer 描述的問題與實際檔案狀態不符（描述的內容在檔案中找不到）
明確說明實際狀態與判斷依據，供用戶決定。

**嚴格限制**：只處理清單中列出的問題，不額外優化。禁止修改清單未提及的任何文字，包括調整周邊句子措辭、拆分段落、修飾風格。
```

## 迴圈終止條件

1. **正常定稿**：reviewer 回「無問題，MOC 可以定稿」→ 結束
2. **收斂到風格微調**：連續 2 輪只剩可選項目 → 詢問用戶是否停（因為已達 ROI 遞減）
3. **不收斂**：超過 5 輪 → 停下，整理各輪變更史交用戶決斷
4. **反覆翻譯**：同一段話被改來改去 → 那段本身就有歧義，重新與用戶討論此段意圖

## 輪數預期

實際經驗（Claude Code Skills MOC 案例）：
- 第 1 輪：4 項（補完整體架構：前言、範例選擇、連結說明、格式微調）
- 第 2 輪：14 項（官方事實校正：token 數字、觸發語法、命名規則、來源註記）
- 第 3 輪：7 項（措辭精確化：敘述歧義、描述機制、句子拆分）
- 第 4 輪：定稿

典型 3-4 輪可收斂。若第 2 輪就跳到純風格微調，MOC 可能本來就不錯；若第 5 輪還有必改，通常是範圍太大，應考慮拆主題。

## Anti-pattern

- **不要用 reviewer 做 writer**：reviewer 只找問題，不自己寫修改版
- **不要讓 fixer 自由發揮**：fixer 只處理清單項目，不得修改清單以外的任何文字，包括周邊句子的措辭調整
- **不要跳過確認**：每輪 review 回報都應讓用戶看到，尤其定稿前
