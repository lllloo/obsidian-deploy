# MOC 結構範本

MOC（Map of Content）是主題入口，不是內容庫。結構應引導讀者快速理解主題與找到細節來源。

## Frontmatter

```yaml
---
title: <主題名稱>
created: <首次建立日期 YYYY-MM-DD>
updated: <今日 YYYY-MM-DD>
tags:
  - <主題類別>      # 如 claude-code / obsidian / ai-tools（即使 MOC 寫在 Cards/，仍用主題 tag 串連，後續升 Topics/ 不需動 tag）
  - <子主題>        # 可選
  - moc             # vault-distill 產生的 MOC 一律加此 tag（state detection 依 moc tag 識別 MOC，不可省略）；若同主題既有其他 MOC 還沒有此 tag，一併補上
---
```

- 不加 `# 標題` heading（Quartz 會從 `title` 自動產生）
- 若尚未完成可加 `draft: true`，定稿後移除
- `source` 欄位通常不需要（MOC 自己就是來源彙整）；若 MOC 主要根據單一官方來源，可加
- **欄位順序 / 白名單**：若要加 `source`, `parent`, `draft` 等可選欄位，順序以 `scripts/vault-schema.mjs` 的 `FIELD_ORDER` 為準；schema 白名單以外的欄位（如 Web Clipper 的 `author`, `description`）一律不可寫入

## 骨架章節（依主題調整順序與取捨）

### 0. 前言（一句話）

用一行列出 MOC 涵蓋的面向（3-5 個），作為目錄指引。
例：「涵蓋是什麼、在生態中的位置、怎麼用、怎麼建、怎麼避坑」。

**前言必須與實際章節對應**，否則產生孤兒敘述。

### 1. 核心概念

- 是什麼（一段定義）
- 關鍵機制 / 內部結構（帶官方引用的數字更佳）
- 怎麼觸發 / 怎麼使用（若適用）

### 2. 對比 / 組合層次

- 與相關技術的比較表（trade-off 而非只列優點）
- 在生態中的位置

### 3. 操作指南 / 怎麼建（若適用）

- 推薦工具（有官方工具就介紹、連結到 vault 內詳細筆記）
- 硬性規則（官方的限制、命名規範、數字上限）

### 4. 常見陷阱

用「徵兆 / 原因 / 解法」三段式。只列操作指南沒講到的反面情境，避免與前面章節重複。

### 5. 來源

分兩層：
```markdown
**官方文件**
- [標題](https://...)
- [標題](https://...)

**影片 / 社群**
- https://...
- https://...
```

若保留影片標題，格式：`- [標題（頻道）](URL)`

## 風格要求

- **繁體中文為主**，技術名詞/品牌名保留英文（Claude Code、SKILL.md、MCP）
- 章節層級：`##` 大章節、`###` 小節、`####` 少用
- **表格**：用於並列對比（四項以內清晰，更多改用清單分組）
- **三段式**：陷阱 / 錯誤 / 問題類型用此結構（徵兆 + 原因 + 解法）
- 數字、比例、規則**必須標來源**（官方 / 社群實測 / 推算）
- wikilink 只連 vault 內確實存在的檔案（Quartz shortest 解析，直接用檔名不含副檔名）
- 外部連結用 `[標題](URL)` 完整 markdown 格式

## 常見錯誤（寫完自檢）

- [ ] 前言列的面向都有對應章節（無孤兒）
- [ ] 沒有指涉「方法論章節」之類已刪除的章節
- [ ] wikilink 檔名正確（檔案存在）
- [ ] 影片 URL 都與 MOC 內容實質相關（無關的移出）
- [ ] 數字（token、%、秒數、行數）都有來源
- [ ] 章節順序：是什麼 → 位置 → 怎麼用 → 怎麼建 → 陷阱 → 來源（不要把怎麼建放在怎麼用之前）
- [ ] 操作指南與陷阱無內容重複
- [ ] 硬性規則清單不要把三件事塞一行
- [ ] frontmatter `updated` 為今日日期

## 長度參考

- 前言：一行
- 每章節：通常 5-30 行，超過 50 行考慮拆子章節
- 全檔案：通常 80-200 行；超過 300 行考慮拆為多個 MOC（如 `<主題>-基礎.md` + `<主題>-進階.md`）
